# UI/UX Audit Findings — 2026-05-15 (Phase 3 of overnight)

## Methodology + tooling used

- **Scope**: 10 routes × 3 viewports = **30 captures** against `https://restoreassist-sandbox.vercel.app`
- **Sandbox tip at audit time**: `95dd34e2` (origin/sandbox HEAD after fast-forward — includes Wave 1: onboarding hotfix + SP-E storage card + SP-A + Seam F + DR/NRPG content)
- **Sandbox availability**: 200 OK on every route (no 5xx, no 404 outside of `_not-found` for unauthenticated dashboard paths which serve the global 404 client-side before redirect)
- **Tools used**:
  - **Playwright (playwright-core 1.60.0, headless chromium-1223)** — navigation, viewport emulation, full-page screenshots, Performance API timing reads, computed-style colour palette extraction
  - **axe-core 4.x via @axe-core/playwright** — WCAG 2.0 A/AA + 2.1 A/AA + 2.2 AA tag set against all 30 captures
  - **Lighthouse 11.11.0** — 9 representative routes (8 mobile + 1 desktop) for Performance / Accessibility / Best-Practices / SEO scores + Core Web Vitals
- **Tools skipped + why**: Did not run Lighthouse for all 30 capture cells (would have been ~10 min and added little — the LH score variance is between routes, not viewports, and axe already covered all 30 for a11y). No Pa11y, no WAVE, no manual screen-reader walkthrough.
- **Auth note**: 5 protected routes (`/setup`, `/dashboard`, `/dashboard/inspections/test-id`, `/dashboard/settings/credentials`, `/dashboard/settings/storage`) all redirect to `/login` for an unauthenticated session — captures + scores are of the login destination, NOT the destination route. The `/api/test/seed-inspection` endpoint requires auth (returns `401 Unauthorized`) so no real inspection id was reachable.
- **Wall-clock**: 30-capture pipeline = 64 s; 9 Lighthouse runs ≈ 9 min serial; total tool wall-clock ≈ 12 min; full audit including analysis ≈ 35 min.

## AI-integration opportunities per route (1 paragraph each)

Each section maps to spec §5 lifecycle hooks (intake → triage → diagnose → estimate → schedule → execute → close).

1. **`/`** — The landing page currently sells features. The AI hook is upstream of signup: a "Get an instant scope read on your last job report" hero CTA that accepts a pasted Xactimate / paper report and returns a structured S500 alignment score. This is **intake-with-friction-removed** — proof of value before the user provides an email.

2. **`/signup`** — Two pain points: (a) the form asks for company details that exist in ABR; (b) the credential dropdown is free-text. Hooks: ABR autofill from ABN (already partly wired via the ABR consumer key per MEMORY); IICRC certificate parsing from drag-and-dropped PDF; auto-detect business address → state for compliance matrix routing.

3. **`/login`** — No AI hook needed at the login _primitive_. But the _abandoned-login fallback_ is an AI opportunity: when login fails 3× in 60 s, show "We can text a magic link to your verified phone" — Twilio SID + AI ranker on which channel succeeded last.

4. **`/setup`** — The wizard is the highest-leverage AI surface: every credential, every storage choice, every team-member invite is a structured-data extraction problem. Hooks: parse uploaded IICRC certs (cert number + expiry + sub-cert list); parse uploaded WHS-RTW / White Card images; auto-fill state-specific licence references from address. Each of these eliminates a "double-handling cliff" (see below).

5. **`/dashboard`** — The dashboard is a **diagnose / triage** surface. AI hooks: a single "What needs attention today?" card that summarises overdue inspections + pending sub-jobs + insurer SLA breaches, generated server-side per session — cached 5 min.

6. **`/dashboard/inspections/[id]`** — The inspection report is the **single most valuable AI surface in the product**. Hooks: photo → moisture-class classifier; photo → contaminant-category classifier (Cat 1/2/3); narrative-from-photos generator (S500-compliant); cost-roll-up suggestion based on similar prior jobs in the same postcode.

7. **`/dashboard/settings/credentials`** — As-is, this is a static CRUD list. AI hooks: certificate-expiry forecasting ("Your IICRC ASD lapses in 47 days — book renewal now"); jurisdictional-matrix delta alerts ("QLD has changed §3.4 of the building code — your stored licence reference is stale").

8. **`/dashboard/settings/storage`** (SP-E) — The Google Drive / OneDrive switch is fine functionally. AI hook: auto-detect from the company domain (`@gmail.com` → suggest Drive; `@outlook.com` / `@*.onmicrosoft.com` → suggest OneDrive); auto-create a parent folder named after the company; AI-classify storage usage at month boundaries.

9. **`/pricing`** — The pricing tiers are static. AI hook: "Estimate my plan" — small calculator widget that infers tier from `(jobs/month, technicians, integrations)` and surfaces the exact line that drives the up-tier — already a known conversion lever in similar SaaS verticals.

10. **`/privacy`** — Largely static legal text. AI hook: **a query box at the top — "Ask a question about how we handle your data"** — backed by a constrained-prompt RAG over the privacy text itself, with strict refusal for anything outside the document scope. This is low-cost and dramatically improves comprehension over a wall of text.

## "Double-handling cliffs"

Places where the user has to re-enter data the system already has (§1.4 success criterion).

1. **ABN on `/signup` → re-entered on `/setup` wizard** — the wizard's "Company details" step doesn't autofill from the signup ABN. Either flow asks for ABN, then the other asks again.
2. **IICRC certificate number** — captured during `/setup` credentials step, then re-entered per-inspection at attestation time. The user has typed the same six-digit certificate number ≥ 3× per inspection.
3. **Address** — collected in signup, re-collected on every inspection's "site address" field. No "use my company address as default" toggle.
4. **State-specific licence reference** — the user types this on the credentials page AND must re-type it inside each inspection's compliance footer.
5. **Storage destination** — set globally on `/dashboard/settings/storage`, but a per-inspection "where do photos go" prompt re-confirms it on every upload. Should be a silent default with an opt-out toggle, not a forced question.
6. **Team-member role** — set at invite, asked again every time that user attempts a transition gated by `canPerformTransition`. The role lookup is a DB read — it does not need to be a UI prompt.
7. **GST + AU locale flags** — hardcoded everywhere but still shows a "Confirm tax setup" step at first claim submission. Redundant.

## Performance hotspots

### Routes failing Core Web Vitals (LCP / CLS / INP)

- **`/` on mobile: LCP 5.2 s** (Good < 2.5 s, Needs Improvement < 4 s, Poor ≥ 4 s) — **POOR**. Desktop LCP is 0.8 s, so the regression is mobile-only and almost certainly a hero-image / above-the-fold-font issue under 3G/4G throttling.
- **`/dashboard/settings/credentials` LCP 4.1 s** — borderline POOR but post-redirect into `/login`, so what's actually being measured is login-page LCP. Login page mobile-only LCP is 3.3 s (NEEDS-IMPROVEMENT).
- **`/pricing` LCP 4.0 s** — NEEDS-IMPROVEMENT. Pricing page is content-heavy with feature cards and ships 30 resources.
- **CLS = 0 across the board** — fine.
- **INP**: not measured (Lighthouse needs user interaction; this audit is page-load only).

### JavaScript bundle size flags

- Per Performance API resource counts: `/` ships **66 resources / 9 kB transfer (HTML)** — the encoded body is small, the perf cost is on the JS/CSS chunks.
- Auth routes (`/login`, `/setup`, post-redirect) ship 35–46 resources / 6 kB HTML.
- Largest gap: `/privacy` ships **19 resources / 11 kB HTML** vs `/` at 66 resources — the marketing pages haul significantly more JS, suggesting hero animations / analytics / motion libs are loaded eagerly when they could be lazy.

### Image optimisation gaps

- All sampled pages use `/_next/image` correctly (no raw `<img>` references in HTML).
- No missing `alt` flagged by axe.
- The mobile-LCP-5.2 s on `/` is not an image-byte problem (transfer = 9 kB HTML) — it's an above-the-fold render-blocking JS / font-loading-strategy issue. Likely fix: preload the hero font subset; mark the hero JS chunk `defer` or move motion-prone components below the fold.

## Severity-graded findings

### P0 — broken / WCAG 2.2 AA critical (must fix before any external launch)

1. **`user-scalable=no` on the root `<meta name="viewport">`** — fails WCAG 1.4.4 across **every page**. One-line fix in the root layout. Blocks visually-impaired users from zooming any form on the site, including credentialing.
2. **"Show password" eye button is 20×20 px on `/signup` + `/login`** — fails WCAG 2.2 §2.5.8 target-size on the two most-used auth surfaces. Risk: serious user-tested mobile-form failures.

### P1 — significant UX friction / WCAG serious

3. **Mobile LCP 5.2 s on `/`** — POOR Core Web Vitals on the marketing landing. Conversion-rate killer; impacts SEO ranking (CWV is a ranking signal). Fix: above-the-fold render-blocking audit on `/` only.
4. **Primary CTA on `/` and pricing buttons have 4.33–4.48:1 contrast** — fail WCAG 1.4.3 for 14 px regular weight text. Affects the highest-leverage commercial elements on the site.
5. **All 5 protected routes hard-redirect to `/login`** without state preservation — the user loses their intended destination (no `?next=` param visible in the redirect URL). After authenticating they land on dashboard root, not where they tried to go. Standard friction.
6. **Brand colour `#D4A574` is declared but unused** — either applied (tag-line / hover) or removed from the canonical palette. Currently a dead token causes brand-guardian noise.
7. **Three different "dark bg" greys in use** — `slate-950`, `neutral-950`, and the declared `#050505` — pick one and tokenise. Brand-consistency drift.

### P2 — polish / suggestion

8. **`/` and `/pricing` have 15 small (< 44 px) tap targets each on mobile** — most are nav links; bumping vertical padding from `py-1` to `py-3` fixes the lot.
9. **`/privacy` footer micro-copy at 4.17:1 contrast** — bump `text-neutral-500` → `text-neutral-400` on the retention paragraph + effective-date line.
10. **`/pricing` uses a 4th colour family (`text-slate-300`) inconsistent with `/privacy`'s `text-neutral-500`** — tokenise body-copy grey across marketing routes.
11. **Cyan-500 accent on `/signup` focus state and 404 page CTA** — unbranded; either adopt as a tertiary or remove.
