# Guideline 4.2 risk assessment for 1.0(7)

## Why this matters

The May 4 rejection on 1.0(6) cited 4.2 with this specific sentence:

> "Including features such as **push notifications, Core Location, or
> sharing** do not provide a robust enough experience to be appropriate
> for the App Store."

Build 1.0(7) added native features that map exactly to Apple's named-and-
rejected categories:

| 1.0(7) addition | Apple's words |
|---|---|
| `@capacitor/local-notifications` (24h follow-up reminder) | "push notifications" |
| `@capacitor/geolocation` (GPS address auto-fill) | "Core Location" |
| `@capacitor/share` (PDF share sheet) | "sharing" |
| `@capacitor/haptics` | not addressed by Apple (additive, low signal) |

So Apple has effectively pre-flagged that this set may not be sufficient.

## What 1.0(7) DOES have going for it

- **Camera + meter-photo OCR** — already shipping, not part of the rejected set, gives the app a real on-device reason to exist (technicians photographing moisture meters in the field, on-device EXIF, OCR feedback).
- **Field-job context** — RestoreAssist is a B2B tool used on real damage sites by technicians, not a consumer browser-replacement.
- **Sign in with Apple + workspace auth** — gating, not browser-equivalent.
- **The four added natives are all wired into the actual workflow** — not bolted on for compliance theatre.

## What 1.0(7) does NOT have

The features Apple typically considers "robust enough" for 4.2:

- **Offline functionality** — does the app work without connectivity? RestoreAssist iOS currently loads `https://restoreassist.app` in a Capacitor WebView; no internet → blank screen.
- **On-device computation** — local image processing, ML models, sensor fusion. Currently camera capture goes through the standard browser file-input → server-side OCR.
- **Hardware integrations** — Bluetooth moisture meters / hygrometers (the real differentiator for restoration field work) were declared in Info.plist but removed for 1.0(7) because the implementation isn't ready.
- **Persistent local state** — offline form drafts, inspection queue, sync-on-reconnect.

## Likely outcomes

**Best case (~50%):** Reviewer sees the four native integrations exercised in a coherent workflow + the sign-in-with-Apple flow + the camera/OCR work that pre-dates 1.0(7) and considers the cumulative experience sufficient. 1.0(7) approved.

**Mid case (~35%):** Reviewer rejects on 4.2 again, citing the same sentence almost verbatim. App Review pushes back on local notifications + GPS + share sheet specifically.

**Worst case (~15%):** 4.2 cleared but a NEW issue surfaces (3.1.1 leftover URL we missed, metadata, screenshots) — back to drawing board for an additional ground.

## Recommended action

1. **Submit 1.0(7) anyway.** It is materially better than 1.0(6). If approved, you ship today. If 4.2 rejected, the rejection letter will tell you specifically what is still missing.

2. **Pre-plan 1.0(8) Bluetooth meter integration** in parallel. If 4.2 rejects 1.0(7), the proven path is hardware integration that can ONLY be done natively — Tramex / Delmhorst / Testo Bluetooth pairing via @capacitor/bluetooth-le. This was scoped earlier (the deleted Info.plist Bluetooth keys are evidence). Bring it back.

3. **In the reply thread (this submission), pre-empt 4.2.** Acknowledge that GPS + Sharing + Notifications alone may not satisfy 4.2 and explain that Bluetooth-meter integration is the next 4.2-level addition. This shows the reviewer you understand the bar — they sometimes give credit for a clear roadmap.

4. **Have a Loom ready.** A 90-second screen recording of you exercising all four native features on a real iPhone, with the reviewer demo account, is the single most effective way to argue 4.2. Apple's 4.2 rejections are much more rare when reviewers can see the app in motion before opening it themselves.

## What this means today

Submit 1.0(7) now. Do not wait. If rejected on 4.2, plan 1.0(8) to add Bluetooth meter pairing — that's the genuinely-different-from-a-browser feature this category of app can defend.
