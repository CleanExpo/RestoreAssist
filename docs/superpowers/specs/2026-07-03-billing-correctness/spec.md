## Problem / intent

RestoreAssist charges real money today, but the paid surface is not safe to take it. The SHIPIT audit (2026-07-03, filed as Linear RA-6929 / RA-6930 / RA-6931) found four defect classes that all end the same way — a restoration-business owner is charged and either receives nothing, receives the wrong thing, or is charged twice:

1. **Two contradictory plan catalogs.** The public/dashboard pricing pages sell **Monthly $99 / Yearly $1188 AUD** [VERIFIED lib/pricing.ts:54-96], while the hard-paywall page every expired-trial user is funnelled to sells a different **STANDARD $99 / PREMIUM $199 / ENTERPRISE** tier catalog [VERIFIED app/billing/upgrade/TierGrid.tsx:14-34] against separate `STRIPE_PRICE_*` env vars [VERIFIED app/api/billing/checkout/route.ts:15-31]. The one page that must convert sells plans the rest of the codebase knows nothing about.
2. **An unpurchasable entitlement.** The floor-plan underlay is gated on a Premium tier [VERIFIED lib/billing/floor-plan-entitlement.ts:14-23], but **no code path anywhere writes `subscriptionTierId`** [VERIFIED grep app/ lib/ = 0 hits], so a $199 PREMIUM buyer can never actually receive what distinguishes the tier.
3. **One-time purchases fulfilled only in the browser.** The Stripe webhook returns early for any non-subscription checkout [VERIFIED app/api/webhooks/stripe/route.ts:397], so lifetime and addon-pack access is granted **only if the buyer's browser returns to the success page** and calls a verify endpoint [VERIFIED app/api/verify-subscription/route.ts:94-118, app/api/addons/verify/route.ts:73-84]. Pay, then close the tab, and money is taken with nothing delivered.
4. **Double-billing.** No checkout route checks whether the user already has an active subscription [VERIFIED app/api/billing/checkout/route.ts:33-95, app/api/create-checkout-session/route.ts:120-135], so an ACTIVE $99 subscriber who clicks another plan gets a second concurrent Stripe subscription and is billed twice.

We need the billing core to be trustworthy: one plan catalog, server-authoritative pricing, webhook-side fulfillment that never depends on the browser, idempotent and replay-safe writes, no double-charge, and every entitlement a plan sells actually provisioned on payment. This must land before any further monetisation (the five $11 add-ons) is built on top of it.

## User stories

- **As a** restoration-company owner (the Business Owner buying the subscription), **I want** the plan I see advertised to be the exact plan I am charged for and the exact plan I receive, **so that** I can trust the product with my company's money and not be quoted one price then billed against a different catalog.
- **As** the paying customer whose trust is the product's foundation, **I want** to be certain that when a payment succeeds I get what I paid for and I am never charged twice for the same thing, **so that** I don't have to chase support for a refund or a missing entitlement after handing over money.
- **As an** admin / office-staff member managing the account, **I want** the account's subscription state, report limits, and add-on packs to reflect exactly what was purchased, **so that** my team isn't blocked mid-job by an entitlement that silently failed to provision.
- **As** internal Ops / the Founder, **I want** every completed Stripe payment to be reconcilable against a fulfillment write with no silent revenue loss and no orphaned subscriptions, **so that** a single config slip or an abandoned browser tab can't leave paid customers unserved or the books un-auditable.

## Acceptance criteria (Given / When / Then)

1. **One catalog.** Given the app is deployed, When a lister enumerates every user-reachable checkout entry point (public pricing, dashboard pricing, the expired-trial hard-paywall), Then every one initiates checkout against a plan from the single source-of-truth catalog, and no page offers a plan/price absent from it. (Executable: a test asserting all checkout CTAs resolve to catalog SKUs; grep for the removed catalog returns 0 sell-able references.)
2. **No client-set price.** Given an authenticated user, When they POST a checkout request with a `priceId`/plan they crafted that is not in the server allowlist, Then the server rejects it (4xx) and creates no Stripe price and no Stripe session. (Executable: unit/integration test posting an unknown price id.)
3. **Browser-independent one-time fulfillment.** Given a user completes a lifetime or addon-pack (payment-mode) checkout, When the `checkout.session.completed` webhook is delivered and the browser never returns to the success page, Then the purchase is fully provisioned (lifetime access / addon reports credited) from the webhook alone. (Executable: fire a synthetic payment-mode `checkout.session.completed` at the handler with no verify call; assert DB state.)
4. **Idempotent fulfillment.** Given any fulfillment-bearing Stripe event, When it is delivered twice with the same event id, Then the entitlement is applied exactly once (no double credit, no duplicate access). (Executable: replay the same event; assert single provisioning + SKIPPED audit row.)
5. **No 400 on legitimate one-time PIs.** Given an addon/lifetime `payment_intent.succeeded` event with no `invoiceId`, When the webhook processes it, Then it returns 2xx and does not error-retry-loop. (Executable: post the event; assert 2xx and correct branch.)
6. **No double subscription.** Given a user whose `subscriptionStatus` is ACTIVE, When they attempt a new subscription checkout, Then no second Stripe subscription is created and they are routed to change-plan / billing portal. (Executable: seed ACTIVE user, attempt checkout, assert no new subscription + expected response.)
7. **Entitlement provisioned by payment.** Given a user buys a plan that grants a specific report limit / feature, When payment completes via the webhook, Then the granting field is written by the fulfillment path and the limit/feature resolver returns the purchased entitlement (not a default because the field was never populated). (Executable: complete checkout, assert entitlement resolver output.)
8. **GST on every path.** Given any checkout path is exercised, When the Stripe session is created, Then it carries `automatic_tax` enabled, GST-inclusive `tax_behavior`, and ABN `tax_id_collection`. (Executable: assert session params on each checkout route, including the previously-bare tier route.)
9. **Scoped refund revocation.** Given a customer with an active subscription refunds a one-time addon charge, When `charge.refunded` is processed, Then the subscription remains ACTIVE. (Executable: fire addon-charge refund event; assert status unchanged.)

## Dependencies & constraints

Standing law that binds this feature:

- **Migrations only, additive.** Schema changes go through `npx prisma migrate dev`; no raw schema edits, no destructive drops without a two-step plan. [CLAUDE.md rule 4; review-dimensions.md #16]
- **New tables require RLS** (RA-6677 gate) — if the fulfillment redesign adds a table, it must ship with row-level security.
- **Subscription allowlist.** Access is gated on `["TRIAL","ACTIVE","LIFETIME"]`; CANCELED / PAST_DUE are blocked at HTTP 402. Note the enum has no `LIFETIME` member — lifetime is modelled as `subscriptionStatus = ACTIVE` + `lifetimeAccess = true` + `subscriptionPlan = "Lifetime"` [VERIFIED prisma/schema.prisma:1304-1310, app/api/verify-subscription/route.ts:98-109]; the catalog redesign must preserve this representation or migrate it deliberately.
- **Auth on every route; webhooks verify signature.** `getServerSession` on user routes; the Stripe webhook verifies `stripe-signature` against `STRIPE_WEBHOOK_SECRET` (a missing secret previously bounced all webhooks — the real incident behind the reconcile script). [CLAUDE.md rule 1; VERIFIED app/api/webhooks/stripe/route.ts:47-75]
- **GST-inclusive pricing, ABN 11 digits, GST 10%.** [CLAUDE.md rule 13]
- **Read before modify** — 120+ Prisma models, 800+ files; never assume a field's writer exists. [CLAUDE.md rule 16]
- **Founder-approved billing model (RA-6920 / RA-6922, "go P0+P1" 2026-07-03):** $99/month AUD BYOK base + five $11/month add-ons via a `FeatureEntitlement` layer. This is NOT the STANDARD/PREMIUM/ENTERPRISE catalog and NOT Monthly-$99/Yearly-$1188. [VERIFIED docs/specs/byok-monetisation-spec.md:22-35]
- **Idempotency + rate-limiting** already wrap the checkout/verify routes (`withIdempotency`, `applyRateLimit`) and must be preserved. [VERIFIED app/api/create-checkout-session/route.ts:42, app/api/addons/checkout/route.ts:38]

## Risks

- **Catalog-direction reversal after build starts (high impact).** If the founder rejects C1's recommendation and wants tier provisioning built instead of deleted, a large share of the plan inverts. *Mitigation:* C1 is gated as a blocking pre-plan decision; no implementation proceeds until it is resolved.
- **Migrating live subscribers during a catalog collapse (data/revenue).** Existing users carry free-text `subscriptionPlan` values ("Monthly Plan", "Yearly Plan", "Lifetime") and, for some, a cast-to-tier assumption on the paywall [VERIFIED app/billing/upgrade/page.tsx:36-39]. A careless collapse could strip access or misprice renewals. *Mitigation:* additive migration + a reconcile/backfill step + explicit grandfathering decision (C4) before any destructive change.
- **Webhook fulfillment redesign changing the money path (high impact).** Moving one-time fulfillment from browser-verify to webhook touches the revenue-critical path; a regression could drop provisioning silently. *Mitigation:* idempotency + StripeWebhookEvent dedupe are already in place [VERIFIED app/api/webhooks/stripe/route.ts:80-115]; require synthetic-event tests (acceptance criteria 3-5) and keep the verify endpoints as a redundant self-heal, not the primary path, during rollout.
- **Config-slip blind spot (silent revenue loss).** A missing `STRIPE_WEBHOOK_SECRET` or unset price env previously desynced all subscriptions; a reconcile script exists but is manual-only. *Mitigation:* fail-closed price allowlist (R3) + treat wiring the reconcile cron as an in-scope safety net or an explicit tracked follow-up. [VERIFIED audit report lines 9, 18]
- **Scope creep into RA-6922 (schedule).** The add-on entitlement layer is tempting to build alongside the catalog fix. *Mitigation:* it is an explicit Non-goal; the catalog must be corrected first as the foundation RA-6922 builds on.
