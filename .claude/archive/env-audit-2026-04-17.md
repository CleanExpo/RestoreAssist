## Risk Flags

### HIGH RISK: Secrets in Multiple Locations

- `.env.local`, `.env.local.pulled`, `.env.vercel.local` contain **identical secrets**
- If one is compromised, all three expose the same credentials
- **Recommendation**: Delete `.env.local.pulled` and `.env.vercel.local` immediately; keep only `.env.local`

### MEDIUM RISK: Production Secrets Scattered

- `.env.production`, `.env.production.local`, `.env.check-prod`, `.env.verify` all contain `STRIPE_SECRET_KEY`, `NEXTAUTH_SECRET`
- Vercel dashboard has authoritative copies, but local files create sync risk
- **Recommendation**: Single source of truth = Vercel dashboard. Delete `.env.production.local`. Use `.env.production` + `.env.local` for local dev only.

### MEDIUM RISK: Docker Secrets Not in Vercel

- `.env.docker.local` defines `POSTGRES_PASSWORD`, `REDIS_PASSWORD` — these are **container-only** and NOT in Vercel
- If someone commits this by accident, database credentials leak
- **Recommendation**: Ensure `.env.docker.local` is in `.gitignore` (verify: `git check-ignore .env.docker.local`). Add pre-commit hook to warn on `POSTGRES_PASSWORD` patterns.

### LOW RISK: Turbo Cache Token Duplicate

- `.env.production` and `.env.verify` include `TURBO_*` and `VERCEL_OIDC_TOKEN`
- Vercel also sets these as build environment variables
- Local copies are redundant but not dangerous if keys rotate
- **Recommendation**: Remove local `TURBO_` vars after consolidation; rely on Vercel dashboard.
